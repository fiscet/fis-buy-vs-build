## What we are building

A free web tool ("Build vs Buy") for **non-technical Italian SME founders**. The user
describes a business process they want to digitize; the tool returns a **side-by-side
comparison** of two paths — buying an existing solution vs building a custom one — across
consistent dimensions, ending with a **contextual lean** (not a hard verdict) based on what
the user described.

It is a **lead magnet**, not a SaaS. Its job is to capture qualified leads (founders
considering digitizing a process) for a custom-MVP services business. The captured lead is
the only real asset of this app — treat lead persistence as critical.

UX requirement: **result must appear in under ~5 seconds**. No live web search at runtime.

### Why a comparison, not a verdict
A one-word verdict from a tool built by someone who sells custom development reads as a sales
pitch and kills trust. A neutral side-by-side comparison feels objective, educates the
founder, and lets the dimensions where custom genuinely wins (process fit, no lock-in, full
control) surface on their own. The soft CTA then lands credibly. The final "contextual lean"
exists only to prevent decision paralysis and give the founder a reason to leave their email.

## Tech stack (do not add to this without reason)

- **Next.js** (App Router) — frontend + API routes
- **Vercel AI SDK** (`generateObject`) — the single structured LLM call
- **Zod** — schema for the LLM output (drives `generateObject`)
- **Turso** (libSQL / SQLite) — lead storage (durable, serverless, no freeze)
- **Resend** — sends the full report by email
- **Upstash Redis** — rate limiting
- **Vercel** — hosting + cron
- Knowledge base lives in a **local JSON file** (`data/kb-seed.json`), read-only at runtime

Use a cost-efficient model for the generation call (configure via env, do not hardcode a
model string deep in the code). One LLM call per user request, maximum.

## Architecture

### Runtime flow (per user request)
1. User submits the form (free-text process description + a few structured fields).
2. **Rate limit check (Upstash) BEFORE any LLM call.** 2 free uses, then email gate.
3. Map the input to a category from the knowledge base (LLM classification or simple match).
4. Read the matching category data from the **local KB JSON**. No live search.
5. Single `generateObject` call: input + KB category data -> full structured report (Zod schema).
6. Render the report with **React components**: a comparison table + the contextual-lean block.
   Never use an LLM to render or format output.
7. On email submission: write the lead to Turso + send full report via Resend.

### Cron flow (monthly — NOT weekly)
1. For each category, run a web search (search API: Tavily or Exa, suited for AI use) for
   current tools + price ranges.
2. LLM synthesizes results into the KB category structure.
3. **Validate before promoting.** Never blindly overwrite: if new output is empty/worse than
   existing data, keep the old data. Write with a timestamp and version field.
4. Update the local KB JSON (versioned). This is the only writer of the KB.

## Data model

### Knowledge base (local JSON, read-only at runtime)
File: `data/kb-seed.json`. Each category holds the raw facts; the comparison itself is
generated by the LLM at runtime from these facts + the user input. Shape per category:

```json
{
  "id": "gestionale-interno",
  "name": "Strumento o gestionale interno su misura",
  "aliases": ["gestionale", "software interno", "..."],
  "description": "…",
  "tension": "alta | media | bassa",
  "commonSolutions": [
    { "name": "…", "type": "saas | onprem | nocode", "priceRange": "…", "bestFor": "…", "limitations": "…" }
  ],
  "typicalCustomCost": { "range": "es. €5.000-15.000", "timeline": "es. 4-8 settimane", "maintenanceNote": "…" },
  "whenBuildWins": "…",
  "updatedAt": "ISO-8601"
}
```

Notes:
- `tension` tells the model how hard to lean: `alta` = build is often a real answer;
  `bassa` = the honest answer is usually "buy, don't build". Respect it — leaning "build"
  on a `bassa` category destroys the tool's credibility.
- `whenBuildWins` is the raw material for the contextual lean toward building.
- Seed prices are plausible placeholders, NOT verified. The cron refines them. Do not present
  seed prices as definitive in user-facing copy until the cron has run.

### Leads (Turso table — durable, critical)
```sql
CREATE TABLE leads (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  email       TEXT NOT NULL,
  company     TEXT,
  category    TEXT,
  input_text  TEXT NOT NULL,
  report_json TEXT NOT NULL,
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  source      TEXT
);
```

## Report schema (Zod — the shape of `generateObject` output)

```ts
const ComparisonRow = z.object({
  dimension: z.enum([
    "costo_iniziale", "costo_ricorrente", "tempo_attivazione", "aderenza_processo",
    "scalabilita", "lock_in", "manutenzione", "controllo_dati",
  ]),
  buy: z.string(),    // how the "buy" option scores on this dimension
  build: z.string(),  // how the "build" option scores on this dimension
  advantage: z.enum(["buy", "build", "pari"]),
});

const ReportSchema = z.object({
  category: z.string(),
  buyOption: z.object({
    summary: z.string(),
    representativeSolutions: z.array(z.object({
      name: z.string(),
      priceRange: z.string(),
      bestFor: z.string(),
      limitations: z.string(),
    })),
  }),
  buildOption: z.object({
    summary: z.string(),
    estimatedCostRange: z.string(),
    estimatedTimeline: z.string(),
    maintenanceNote: z.string(),
  }),
  comparison: z.array(ComparisonRow), // one row per dimension, same axes for both paths
  contextualLean: z.object({
    tipsToward: z.enum(["buy", "build", "hybrid", "depends"]),
    reasoning: z.string(),  // tied to what the founder actually described
    nextStep: z.string(),   // soft CTA
  }),
  confidenceNote: z.string(), // honest note on data freshness / limits
});
```

## Rendering rules (React only)

- Render `comparison` as a **side-by-side comparison table**: one row per dimension, two
  columns (Compra / Sviluppa), with a visual marker for `advantage`.
- Render `buyOption.representativeSolutions` as a short list (name, price range, fit, limits).
- Render `buildOption` as a compact card (cost range, timeline, maintenance note).
- Render `contextualLean` as a closing block: the lean, the reasoning, the next step (CTA).
- Keep it neutral in tone. The comparison must not look like it is steering toward building.
- **Never** use an LLM to produce or format any of this. It is deterministic React over the
  structured object.

## Hard rules (guardrails)

- **Never** use an LLM for rendering or formatting. React only.
- **Never** run a live web search on the user request path. KB JSON only.
- **Always** rate-limit BEFORE the LLM call, not after.
- The cron must validate search output before overwriting the KB. Good data is sacred.
- KB JSON is read-only at runtime; the cron is its only writer.
- Lead writes must succeed durably (Turso). Never store leads in a local file.
- One LLM call per user request. If you think you need more, stop and ask.
- Respect each category's `tension`: do not lean "build" on a `bassa` category.
- Keep secrets in env vars. Never commit keys.

## Build plan (do this in phases, commit per phase)

1. **Scaffold**: Next.js App Router, install deps, `.env.example` with all keys, README.
2. **Schemas + seed KB**: implement `ReportSchema`; wire in `data/kb-seed.json` (5 categories

3. **Core API route**: input -> category match -> read KB -> `generateObject` -> return report.
   Test with the seed KB.
4. **Frontend**: input form + comparison-table rendering + contextual-lean block. Get a working
   end-to-end demo here.
5. **Lead capture**: Turso write + Resend email + the 2-use email gate.
6. **Rate limiting**: Upstash, before the LLM call.
7. **Cron**: monthly search per category, validation, versioned KB write. Only after 1-6 work.
8. **Deploy**: Vercel, configure cron + env.

Get phases 1-4 working as a demo before touching infra (5-7). Avoid premature infrastructure.

## Categories in v1 (confirmed)

All five are `tension: alta` (build is often a real answer — these generate the service leads):
1. gestionale-interno — Strumento o gestionale interno su misura
2. prenotazioni-appuntamenti — Sistema di prenotazioni o appuntamenti
3. portale-clienti — Portale clienti o area riservata
4. ecommerce-custom — E-commerce: funzionalità oltre la piattaforma standard
5. integrazione-sistemi — Integrazione tra sistemi esistenti

Later: add 2-3 `tension: bassa` categories (e.g. fatturazione elettronica, email marketing)
on purpose. When the tool honestly says "just buy this, don't build", its "build"
recommendations become credible.

## Open decisions (ask the user, do not assume)

- The structured input fields on the form (beyond the free-text description): e.g. team size,
